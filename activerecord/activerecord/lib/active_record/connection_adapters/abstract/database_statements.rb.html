173 # Runs the given block in a database transaction, and returns the result
174 # of the block.
175 #
176 # == Nested transactions support
177 #
178 # Most databases don't support true nested transactions. At the time of
179 # writing, the only database that supports true nested transactions that
180 # we're aware of, is MS-SQL.
181 #
182 # In order to get around this problem, #transaction will emulate the effect
183 # of nested transactions, by using savepoints:
184 # https://dev.mysql.com/doc/refman/5.7/en/savepoint.html
185 # Savepoints are supported by MySQL and PostgreSQL. SQLite3 version >= '3.6.8'
186 # supports savepoints.
187 #
188 # It is safe to call this method if a database transaction is already open,
189 # i.e. if #transaction is called within another #transaction block. In case
190 # of a nested call, #transaction will behave as follows:
191 #
192 # - The block will be run without doing anything. All database statements
193 # that happen within the block are effectively appended to the already
194 # open database transaction.
195 # - However, if +:requires_new+ is set, the block will be wrapped in a
196 # database savepoint acting as a sub-transaction.
197 #
198 # === Caveats
199 #
200 # MySQL doesn't support DDL transactions. If you perform a DDL operation,
201 # then any created savepoints will be automatically released. For example,
202 # if you've created a savepoint, then you execute a CREATE TABLE statement,
203 # then the savepoint that was created will be automatically released.
204 #
205 # This means that, on MySQL, you shouldn't execute DDL operations inside
206 # a #transaction call that you know might create a savepoint. Otherwise,
207 # #transaction will raise exceptions when it tries to release the
208 # already-automatically-released savepoints:
209 #
210 # Model.connection.transaction do # BEGIN
211 # Model.connection.transaction(requires_new: true) do # CREATE SAVEPOINT active_record_1
212 # Model.connection.create_table(...)
213 # # active_record_1 now automatically released
214 # end # RELEASE SAVEPOINT active_record_1 <--- BOOM! database error!
215 # end
216 #
217 # == Transaction isolation
218 #
219 # If your database supports setting the isolation level for a transaction, you can set
220 # it like so:
221 #
222 # Post.transaction(isolation: :serializable) do
223 # # ...
224 # end
225 #
226 # Valid isolation levels are:
227 #
228 # * <tt>:read_uncommitted</tt>
229 # * <tt>:read_committed</tt>
230 # * <tt>:repeatable_read</tt>
231 # * <tt>:serializable</tt>
232 #
233 # You should consult the documentation for your database to understand the
234 # semantics of these different levels:
235 #
236 # * https://www.postgresql.org/docs/current/static/transaction-iso.html
237 # * https://dev.mysql.com/doc/refman/5.7/en/set-transaction.html
238 #
239 # An ActiveRecord::TransactionIsolationError will be raised if:
240 #
241 # * The adapter does not support setting the isolation level
242 # * You are joining an existing open transaction
243 # * You are creating a nested (savepoint) transaction
244 #
245 # The mysql2 and postgresql adapters support setting the transaction
246 # isolation level.
247 def transaction(requires_new: nil, isolation: nil, joinable: true)
248 if !requires_new && current_transaction.joinable?
249 if isolation
250 raise ActiveRecord::TransactionIsolationError, "cannot set isolation when joining a transaction"
251 end
252 yield
253 else
254 transaction_manager.within_new_transaction(isolation: isolation, joinable: joinable) { yield }
255 end
256 rescue ActiveRecord::Rollback
257 # rollbacks are silently swallowed
258 end
